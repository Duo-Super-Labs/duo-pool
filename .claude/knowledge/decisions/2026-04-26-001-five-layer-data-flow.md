# ADR-001: 5-Layer Data Flow inspired by duo-admin

**Status:** Accepted
**Date:** 2026-04-26
**Context:** DuoPool live demo for *AI Amplifica. O Processo É Seu.* (Univali talk)

## Context

Need an architecture that:
1. Forces strict separation between data, contracts, server, and frontend
2. Lets the live `/duo.exec` demo prove that pre-defined structure + types make AI-generated code respect the architecture
3. Scales conceptually to a real product without ceremony for a small demo

The talk's pedagogical goal hinges on the architecture being **physical, not conventional**. Convention can be ignored; structure cannot.

## Decision

Adopt the 5-Layer API Data Flow from `~/workspace/work/gebra/duo-admin/CLAUDE.md`:

```
L1: packages/database/src/schema/*.ts        Drizzle table definitions
L2: packages/database/src/schema/zod.ts      drizzle-zod auto-derived validators
L3: packages/database/src/query/*.ts          Pure functions: fn(db, input)
L4: packages/contracts/src/*.ts               oRPC contracts (frontend-importable)
L5: packages/api/src/modules/*/procedures/*.ts  publicProcedure.handler()
```

Drop from the duo-admin reference: auth, RBAC, tenant middleware, jobs, mail, storage, KV, logs, mocks. Keep: layered flow, module anatomy in `apps/web/modules/`, drizzle-zod single source of truth, NEVER DO list.

## Consequences

- **(+)** Frontend (`apps/web`) imports `@duopool/contracts` for types — cannot accidentally pull `pg`, Drizzle, or query functions into the client bundle
- **(+)** `polls.vote` live demo will gain weight: code generated by `/duo.exec` must touch 5 distinct files (L3, L4, L5, frontend api hook, frontend UI). Skipping a layer breaks `tsc`
- **(+)** Mirroring a real production starter makes the demo feel "real" instead of toy
- **(−)** More packages = more boilerplate for a 3-table app. Acceptable: the architecture itself IS the lesson
- **(−)** Anyone reading the code expecting tRPC/Express/etc. needs to learn oRPC — mitigated by the duo-admin pattern being explicit and documented
